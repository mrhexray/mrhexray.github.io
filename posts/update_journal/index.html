On the side I also learned that signals are used to communicate with processes and we can use user defined signals as well. However I’ll take a look later.

Then in that tutorial explained why we shouldn’t use printf or fprintf inside the handlers. Its because signal handliers are asynchronous and write system call is async signal safe.

I digged a bit about asynchronous function and why printf may crash our program.

I found on internet that asynchronous means “unpredictable timing”, which I didn’t understand ultimately.

Please clarify in simple terms.

But I found the reason why printf is unsafe to use in handlers. Printf uses stdout buffer to put string and its not flushed to terminal instantly, instead it is line buffered, which mean unless it hits newline char, it remains in buffer and not sent to terminal to show.

And it preserves the content in buffer across multiple calls to printf to use the write sys call efficiently. To do so it uses locking to that buffer and hence we find the issue.

if main’s printf starts executing and it firstly locks the stdout buffer, then while execution signal fault occurs and our handler runs. Now if our signal handler also has printf call then it will try to aquire lock on same stdout buffer, but its already locked by main’s printf, and so it will wait for it to release the lock, but since main is freezed until the handler is completely executed, it waits indefinetly in deadlock.

I got it why printf use can cause these issues due to its locking nature on stdout. But I am not sure about what do we mean by asynchronous exactly, please explain clearly with examples.

While learning about inner working of printf (just a little), I found that stdout, stdin, stderr are file streams and they are mapped to a region called mapped memory, which lies between heap and stack. I furthur investigated and found that this region (in modern systems) grows downwards the same direction as of stack.

I also learnt about the volatile keyword, which tells compiler to not cache the variable in register. We need this behaviour because if we are using the variable in handler we won’t want it to be cached in register (by compiler optimization).

for example

int flag = 0; // Not volatile
while (flag == 0) {
    // Do some work that doesn't touch 'flag'
}

here compiler will see that we are not modifying the flag in loop, so instead or reading flag from memory for n number of times for loop condition, why not add extra copy instruction to copy the flag to a register, and then read from there it will be much faster.

But if durint loop its interrupted and our handler runs, and then it modifies the flag value inmemory, the loop will have no idea about this and later when handler is done, and main continues, it will be executing incorrectly with old value of flag.

src/signals.c :

#include <stdio.h>
#include <stdlib.h>
#include <signal.h>
#include <unistd.h>
#include <string.h>

// Global storage to restore original behavior later if needed
struct sigaction old_action;

// The custom signal handler function
void segv_handler(int signum, siginfo_t *info, void *context) {
    // 1. CRITICAL: Use only async-signal-safe functions like write()
    const char msg[] = "CRITICAL: Caught SIGSEGV (Segmentation Fault)!\n";
    write(STDERR_FILENO, msg, sizeof(msg) - 1);

    // 2. Optional: Extract the exact memory address that caused the fault
    if (info != NULL) {
        // Note: Formatting strings directly in handlers using printf is unsafe.
        // This is a minimal, raw illustration using write.
        write(STDERR_FILENO, "Faulting address: \n", 19);
        // system tools or logging would usually resolve info->si_addr
    }

    // 3. Reset to default handler so the next fault causes a core dump
    sigaction(SIGSEGV, &old_action, NULL);

    // 4. Terminate process. Returning from a SIGSEGV handler without 
    // fixing the memory layout causes an infinite loop of faults.
    _exit(EXIT_FAILURE); 
}

int main() {
    struct sigaction sa;

    // Clear the structure completely
    memset(&sa, 0, sizeof(sa));

    // Assign the 3-argument handler function
    sa.sa_sigaction = segv_handler;

    // Use SA_SIGINFO to tell the system to pass advanced info to the handler
    sa.sa_flags = SA_SIGINFO;

    // Block all other standard signals while this handler runs
    sigemptyset(&sa.sa_mask);

    // Intercept SIGSEGV and register our custom handler
    if (sigaction(SIGSEGV, &sa, &old_action) < 0) {
        perror("Error registering sigaction");
        return 1;
    }

    printf("SIGSEGV handler installed. Triggering crash...\n");

    // Intentionally cause a segmentation fault (dereferencing a NULL pointer)
    int *ptr = NULL;
    *ptr = 42; 

    return 0;
}
